# 知乎回答页 A2A Demo 设计规格

## 结论

当前项目里的 A2A 形态需要重做。旧版本是“用户 Agent 和答主 Agent 模拟对话”，本质更像需求访谈模拟器；我们讨论后确认，比赛 Demo 应该变成一个“知乎回答写作现场”的 A2A 创作闭环。

新目标是：做一个独立的 localhost 页面，复刻知乎写回答页的演示场景。页面不需要完全真实复刻知乎所有功能，但关键演示链路要可信：知乎开放接口真实调用，大模型真实调用，A2A 在创作助手里服务问题意图理解、直答参照、答主画像化策略和发布后互动承接；发布完成后进入单独的回答详情/结果页，评论区互动在发布后的页面发生。

## 产品定位

产品不是“AI 代写回答”，而是“答主写回答时的多 Agent 创作协同与发布后互动承接”。

它要帮助答主在发布前理解：

- 我这个答主基于自己的关注流、关注关系和内容视野，适合从什么角度回答。
- 这个提问者可能真正关心什么，可能讨厌什么表达。
- 全网和知乎语境里哪些观点已经被说烂，哪些角度有争议，哪些材料可以参考。
- 发布后，在回答详情页中，提问者侧 Agent 会真实参与评论区互动，继续追问、认可、反驳或补充。

核心表达是：AI 辅助，人为主导。Agent 可以分析、互搏、提建议，并在发布后作为提问者侧参与互动承接，但不替答主直接发布完整正文。

## 页面形态

采用独立网页 1:1 复刻知乎回答编辑页，运行在 localhost。

不做真实浏览器插件，也不嵌入真实知乎页面。这样最适合比赛演示：稳定、可控、路径短，同时能展示完整产品体验。

这里的“1:1 复刻”指复刻比赛演示所需的关键页面结构、视觉层级和交互位置，而不是复刻知乎所有产品能力。页面要让评委第一眼就能看出：这是知乎问题页里的回答编辑器，A2A 功能嵌在原生创作助手里。

### 页面复刻范围

需要重点复刻截图中的这些区域：

- 顶部导航栏：知乎 Logo、关注、推荐、热榜、专栏、圈子、付费咨询、知学堂、搜索框、直答、加号、消息、私信、创作中心、头像。
- 问题头部：话题标签、问题标题、问题描述、关注者数、被浏览数。
- 问题操作区：关注问题、编辑回答、邀请回答、好问题、添加评论、分享、更多。
- 顶部消息通知：消息、私信、创作中心和头像；点击消息后展示类似截图右侧的通知下拉层。
- 编辑器容器：灰色页面背景中的白色编辑卡片。
- 编辑器标题区：图文回答 tab、全屏编辑入口。
- 编辑器工具栏：撤销、重做、清除格式、标题、加粗、斜体、列表、目录、引用、分割线、代码块、注释、图片、视频、链接、收益、公式、表格、附件、导入、草稿备份、更多、创作助手。
- 正文编辑区：大面积空白输入区域，placeholder 类似“写回答...”。
- 右侧创作助手区：在编辑器内部右侧展开，不脱离编辑器。

其他知乎页面功能可以静态展示，不需要真实可用。

### 参考截图

实现时以用户提供的知乎截图作为主要视觉参考：

- 普通问题页编辑态截图：用于复刻顶部知乎导航、问题头部、问题操作区、编辑器卡片和右侧创作助手入口。
- 全屏编辑态截图：用于复刻沉浸式编辑器、顶部“知乎｜图文回答”标题栏、完整工具栏、右侧创作助手展开区、底部发布栏。
- 通知下拉截图：用于复刻顶部消息入口、红点数字、通知 tab 行、互动消息列表和底部“设置/查看全部通知”。

这些截图的作用是视觉基准。后续实现如果在“通用知乎风 UI”和截图细节之间取舍，应优先贴近截图。

### 视觉设计原则

视觉 thesis：整体像知乎原生编辑器一样克制、干净、工具化；我们的 A2A 能力像“创作助手新增的一项高级能力”，而不是外部强行塞进来的 AI 产品。

内容 plan：

- 编辑区负责写作，不承担解释产品能力。
- 创作助手首页负责保持知乎原生感。
- A2A 子面板负责展示画像、互搏和写作策略。
- 发布后的回答详情页负责展示评论区社交闭环。
- 顶部通知负责承接发布后的互动提醒，让 Agent 生成的提问者评论像真实知乎互动一样进入消息体系。

交互 thesis：

- 从“创作助手”进入 A2A，使用轻量切换，不跳页。
- 画像、研究、互搏结果逐步出现，避免长时间空 loading。
- 普通模式和全屏模式共享同一份 A2A 状态，切换时不重新请求。
- 发布后互动同时体现在“评论区内容”和“顶部通知提醒”两处，但通知只做入口和提醒，不承载 Agent 分析。

### 全屏编辑

编辑器必须包含“全屏编辑”能力。

要求：

- 普通模式下，编辑器位于问题页下方，保留顶部导航、问题头部和页面背景。
- 点击“全屏编辑”后，进入沉浸式编辑状态。
- 全屏编辑状态中，编辑器应占据主要视口空间，工具栏固定在顶部，正文编辑区扩大，右侧创作助手仍可展开。
- 全屏状态要保留退出入口，点击后回到普通问题页编辑状态。
- A2A 创作助手在普通模式和全屏模式中都能工作，状态不能丢失。
- 全屏模式不需要真实调用浏览器 Fullscreen API，可以用应用内全屏布局模拟；视觉上要像知乎编辑器进入全屏。

全屏编辑态需要贴近用户提供的第二张截图，具体包括：

- 顶部白色标题栏左侧显示蓝色“知乎”Logo，中间有竖线分隔，旁边是“图文回答”标题和蓝色下划线。
- 顶部标题栏右侧显示用户头像。
- 第二行是横向编辑器工具栏，图标和文字上下排列，整体横跨页面宽度。
- 页面背景为浅灰，中央为白色编辑工作区。
- 左侧正文区显示问题标题，例如“大学期间如何平衡学业、社交与个人成长？”，右侧有“问题描述”下拉文字。
- 标题下方有细分割线，正文 placeholder 为“写回答...”。
- 右侧创作助手面板在编辑区内部展开，左边有竖向分割线，面板标题为“创作助手”，右上有固定/关闭类小图标。
- 创作助手初始态保留内容检测、智能排版、提取导语、AI 配图、更多功能这些静态功能块。
- 底部固定发布栏显示“发布设置”“字数：0”“Markdown 语法输入中”和右侧“发布回答”按钮。
- 全屏态下 A2A 核心能力要嵌入这个创作助手面板，而不是替换掉整页编辑器。

### 创作助手入口位置

A2A 功能要放在知乎编辑器右侧已有的“创作助手”区域里，而不是做成页面外部的独立大面板。

参考截图中的位置：

- 编辑器上方保留知乎式工具栏，例如撤销、标题、加粗、斜体、列表、引用、图片、视频、链接、公式、表格、导入、草稿备份、更多等。
- 工具栏最右侧保留“创作助手”入口图标和文字。
- 点击或默认展开后，右侧出现创作助手面板。
- A2A 功能作为创作助手里的核心能力呈现，可以命名为“创作协同”“A2A 写作协同”或“回答前后反馈”。
- 原有创作助手里的“内容检测、智能排版、提取导语、AI 配图、更多功能”等可以静态展示，用于贴近知乎编辑器真实氛围。
- 真正可交互的比赛核心功能放在创作助手面板内部，不另起一个独立产品入口。

因此，Demo 的视觉重点应是：用户正在知乎写回答，右侧创作助手自然展开，并在其中完成画像分析、A2A 互搏和写前建议。发布后不再停留在创作助手里展示反馈，而是进入发布后的回答详情页。

### 创作助手改造方案

截图中的创作助手可以稍作修改加入我们的能力，原则是“小改原生，大显 A2A”。

默认打开创作助手时，仍然像知乎原生面板：

- 顶部标题仍为“创作助手”。
- 保留欢迎语，例如“Hello，欢迎使用知乎创作助手，你可选择以下功能辅助创作更优质的内容”。
- 保留原功能卡片：内容检测、智能排版、提取导语、AI 配图、更多功能。
- 在这些功能卡片上方或第一行新增一个更醒目的 A2A 入口卡片。

A2A 入口卡片建议：

- 名称：`A2A 创作协同`。
- 副文案：`基于你的关注流、提问者画像和全网语境，对齐问题意图、参考语境和发布后的互动承接`。
- 状态文案：未开始时显示 `开始分析`；分析中显示 `画像生成中`；完成后显示 `策略已就绪`。
- 视觉：使用知乎蓝作为主强调色，但整体仍是浅灰卡片，不使用夸张渐变。
- 尺寸：可以做成横向满宽卡片，放在原功能网格上方；也可以占两格宽，区别于普通功能。

点击 A2A 入口后，不跳出创作助手，而是在同一右侧面板中切换到 A2A 子面板。

### A2A 子面板细节

A2A 子面板保留创作助手外壳：

- 面板顶部仍显示“创作助手”。
- 次级标题显示 `A2A 创作协同`。
- 提供返回按钮，例如 `返回全部功能`。
- 右上角仍保留固定/关闭图标，贴近截图。

子面板内部使用三段式结构：

1. 写前对齐
2. 参考素材
3. 互搏过程

可以使用紧凑的分段控件或 tab，不要做成大面积卡片堆叠。

写前对齐区包含：

- 进度条或步骤列：`答主画像`、`提问者画像`、`全网语境`、`三方互搏`、`策略生成`。
- 答主画像摘要：2-3 个短标签，例如关注领域、表达偏好、受众期待。
- 提问者画像摘要：展示当前 Demo 提问者画像，例如“焦虑求助型”，但视觉颜色必须符合知乎审美。
- 全网语境摘要：展示已说烂角度、可参考角度、风险点。
- 互搏日志：默认折叠，只展示两轮结论，点击展开看 Agent 观点。
- 写作策略：置顶展示最有用的 3 项，例如“建议切入角度”“别踩的坑”“发布后最可能被追问”。

右侧画像视觉约束：

- 不使用黄色作为“焦虑求助型”主色。黄色在知乎编辑器里显得突兀，也容易像警告态。
- 提问者画像使用知乎蓝的浅色体系，例如浅蓝背景、蓝灰文字、蓝色细边框。
- 风险、禁忌、争议点可以使用低饱和红色点缀，但不要整块红底。
- 答主画像使用中性蓝灰或浅绿色点缀，表达“可用素材/个人优势”。
- 全网语境使用灰蓝或浅紫蓝点缀，表达“外部参考”，避免多彩卡片堆叠。
- 标签风格应接近知乎：浅底、细边框、4-6px 圆角、字号 12-13px。

参考素材区包含：

- 知乎相似问题、相似回答、高赞回答链接。
- 全网参考链接，如果有真实搜索结果。
- 每条素材展示标题、来源、可借鉴点和风险。

互搏过程区包含：

- 两轮三方互搏日志。
- 默认展示摘要，支持展开查看 Agent 具体观点。
- 不在这里展示发布后反馈或评论区输入框。

### 创作助手面板尺寸与布局

普通编辑态：

- 创作助手在编辑器右侧展开，宽度建议 360-420px。
- 左侧编辑区和右侧创作助手之间使用 1px 浅灰分割线。
- 面板内部滚动，不影响左侧正文编辑。

全屏编辑态：

- 右侧创作助手宽度贴近截图比例，建议约 400-430px。
- 工具栏、正文区、创作助手三者布局稳定，不能因为 A2A 内容展开导致工具栏换行或编辑区抖动。
- 如果 A2A 内容过长，只滚动面板内部，底部发布栏保持固定。

响应式：

- 比赛 Demo 以桌面视口为主。
- 窄屏下可以将创作助手改为抽屉，但第一版优先保证桌面截图级复刻。

### 创作助手微交互

- 点击“创作助手”工具栏入口：打开/收起右侧面板。
- 点击 `A2A 创作协同`：进入 A2A 子面板并启动或恢复分析。
- 切换普通/全屏编辑：保持当前 tab、已生成策略和评论上下文。
- 分析过程中：每个阶段完成后即时更新，不等全部完成才展示。
- 生成策略完成后：A2A 入口卡片回到首页时显示 `策略已就绪`，便于用户知道结果可看。
- 点击发布后：离开编辑器，进入发布后的回答详情/结果页。
- 发布后的评论区互动：在回答详情/结果页发生，不在创作助手面板内发生。

页面需要模拟这些关键区域：

- 顶部知乎式导航栏：Logo、搜索框、创作入口、用户头像等可以静态，但布局要贴近截图。
- 问题区：问题标题、问题描述、关注人数、浏览数据、邀请回答按钮等可以静态。
- 已有回答区：展示 1-2 条静态高赞回答摘要，用于制造真实语境。
- 我的回答编辑器：1:1 复刻截图中的图文回答编辑器，答主可以手动输入，也可以点击插入 Demo 草稿。
- 右侧创作助手面板：在知乎编辑器原生“创作助手”位置内展示答主画像、提问者画像、全网研究、Agent 互搏、写作建议。
- 模拟发布区：点击“发布回答/模拟发布”后，跳转到单独的回答详情/结果页。
- 发布后页面：展示已发布回答和评论区，提问者 Agent 的评论互动在这里发生。

## 真实与模拟边界

### 必须真实

这些部分必须使用真实调用，不能只做静态假数据：

- 知乎比赛开放接口调用。
- 答主画像的数据获取与分析。
- 大模型调用。
- A2A 互搏结果生成。
- 写前策略建议生成。
- 答主模拟发布后的提问者反馈生成。
- 评论区多轮互动里的 Agent 回复。

### 可以模拟

这些部分为了演示稳定性，可以静态或半静态：

- 知乎页面外壳。
- 当前问题内容。
- 问题作者信息。
- 已有回答摘要。
- 发布动作本身。
- 静态评论区样式。
- 全网研究素材的一部分，如果没有稳定全网搜索 API，可以用静态研究片段。

## 真实性边界

这里的“真实”指链路和交互真实，不是说 Demo fixture 一定来自真实提问者账号。

- 答主侧数据真实：来自登录人的知乎开放接口数据。
- 提问者侧数据可以是比赛 Demo fixture：用于构造一个可控的提问者上下文。
- 提问者侧 Agent 的互动真实：它不是“预测别人会怎么说”的报告，也不是静态脚本，而是在系统中作为 A2A 参与方，基于问题、画像、答主回答和评论上下文真实调用大模型生成评论。
- 发布后的评论区互动是真实运行的多轮 Agent 交互，只是提问者侧身份数据可以由 fixture 提供。

因此文案上避免使用“生成提问者侧互动”作为核心表达，优先使用“提问者侧 Agent 真实参与评论区互动”。

## 数据来源

### 答主数据

答主就是当前登录人，也就是“我”。

后端用已配置的知乎 token 调用比赛接口，构建我的个性化内容视野：

- `GET /openapi/feed/following`：获取关注人动态与内容流。
- `GET /openapi/user/following`：查询关注列表。
- `GET /openapi/user/followers`：查询粉丝列表。

这些数据用于分析：

- 我关注的创作者和话题。
- 我常看的内容领域。
- 我的社交关系和受众结构。
- 我可能适合的回答角度。
- 哪些表达不能伪造成“我的亲身经历”。

### 提问者数据

提问者不使用真实 user_id，先用模拟画像。

原因：当前 Demo 重点是完整展示写作页 A2A 闭环，真实提问者数据不稳定，而且用户已经确认“提问者可以模拟一下信息”。

提问者画像提供三个模板：

- 焦虑求助型：想要具体建议，讨厌空泛鸡汤。
- 理性质疑型：会追问证据、数据、边界条件。
- 经验补充型：自己有经历，希望答主回应细节。

默认选择焦虑求助型。该模板会交给大模型扩写成可互动的提问者侧 Agent 画像。这里的数据可以是 Demo fixture，但生成和互动链路必须真实运行。

### 全网和知乎语境

全网 Agent 负责结合：

- 当前问题。
- 知乎搜索结果，如果现有知乎搜索接口可用。
- 知乎开放接口返回的相关回答和相关问题。
- 全网搜索结果，如果比赛环境或后端能力允许。
- 静态全网研究片段，仅作为接口失败时的兜底。
- 已有回答摘要。

输出应该包括：

- 常见观点。
- 已经过度使用的角度。
- 争议点。
- 可引用的事实或材料。
- 不建议碰的风险表达。
- 参考素材链接：来自知乎相似回答、知乎相关问题、全网相似内容。

参考素材要求：

- “参考素材”和“高赞思路”不能只写成一句泛泛建议，要尽量给出可点击链接。
- 知乎侧优先使用比赛提供的知乎接口或现有知乎搜索接口，抓取几个相似问题、相似回答、高赞回答摘要。
- 全网侧可以由大模型基于搜索结果摘要做筛选和归纳；如果没有真实全网搜索 API，就明确标记为“静态演示素材”。
- 每条参考素材至少包含：标题、来源、链接、为什么值得参考、可借鉴角度、不能照搬的点。
- 大模型负责从知乎回答和全网结果中提炼“高赞思路”，但必须保留来源链接，避免变成无来源观点。

## Agent 设计

### 答主画像 Agent

输入：

- 我的关注流。
- 我的关注列表。
- 我的粉丝列表。
- 当前问题。
- 可选：编辑器中已有草稿。

输出：

- 我的内容兴趣和关注领域。
- 我的受众可能期待什么。
- 我适合的表达方式。
- 我不应该伪造的个人经验边界。
- 当前问题是否适合我回答，以及适合从哪里切入。
- 可用于当前回答的真实素材线索，例如关注流中常看的主题、关注对象的专业领域、粉丝结构暗示的读者预期。

画像质量要求：

- 画像内容必须来自真实接口数据的归纳，而不是大模型凭空编造。
- 输出要对创作者有用，不能停留在“你关注科技、教育”等泛泛标签。
- 每个画像结论尽量带证据，例如“关注流中多次出现计算机专业/大学规划内容，因此适合从学习路径和踩坑经验切入”。
- 不允许生成答主没有提供过的个人经历，只能给“可写方向”和“素材边界”。

### 提问者侧 Agent

输入：

- 预设提问者模板。
- 当前问题。
- 静态问题描述。
- 后续评论区上下文。

输出：

- 提问者真实困惑。
- 提问者可能讨厌的回答方式。
- 提问者可能希望答主补充的细节。
- 发布后作为评论区参与方发出的第一条评论。
- 评论区后续追问、认可、反驳或补充。

### 全网研究 Agent

输入：

- 当前问题。
- 知乎搜索结果。
- 知乎相关回答/问题链接。
- 全网搜索结果链接，如果可用。
- 静态全网参考素材。
- 已有回答摘要。

输出：

- 已有观点地图。
- 高赞回答常用套路。
- 容易翻车的判断。
- 可以参考但不能照抄的材料。
- “这个回答要有公共参考价值，需要补上什么”。
- 参考素材列表。
- 高赞思路列表。

每条参考素材结构：

- `title`：素材标题。
- `source`：知乎回答、知乎问题、全网文章等。
- `url`：可点击链接。
- `summary`：一句话摘要。
- `useful_for`：适合借鉴到回答里的哪一部分。
- `risk`：照搬或引用时的风险。

### 互搏编排 Agent

不是让三个 Agent 无限聊天，而是让它们短平快地互相校正。

第一版固定互搏两轮：

- 第 1 轮：三个 Agent 各自给出立场。
- 第 2 轮：每个 Agent 指出另一个视角中的一个风险、盲点或反例。

互搏结束后输出：

- 答主是否适合回答这个问题。
- 推荐切入角度。
- 不推荐切入角度。
- 写作结构建议。
- 需要引用或避开的材料。
- 对提问者最可能有效的表达方式。
- 发布后最可能被追问的问题。

### 发布后页面互动 Agent

发布后互动仍然是 A2A 的关键，但它不属于编辑器右侧创作助手。

答主点击“发布回答/模拟发布”后：

- 前端切换到单独的回答详情/结果页。
- 页面展示已发布回答正文。
- 评论区成为主要互动区域。
- 读取答主的回答草稿。
- 读取提问者画像。
- 读取写前互搏结论。
- 大模型真实生成提问者侧 Agent 的第一条评论，并插入发布后页面的评论区。

评论区多轮互动：

- 答主可以在发布后页面的评论区手动回复。
- 提问者侧 Agent 会根据答主回复继续生成下一条追问、反驳、认可或补充。
- Demo 中最多演示 2-3 轮，避免等待太久。
- 不需要在创作助手里展示“发布反馈区”或“答主回应建议区”。

### 与知乎直答结合

评论区的 AI 能力可以借鉴知乎“直答”的产品语义，但不能直接做成另一个聊天助手。

角色拆分：

- `DirectAnswerAgent`：中立参照层，不代表任何用户，不在评论区作为账号露出。
- `AskerAgent`：提问者/读者侧角色，负责把中立缺口转成像真人评论的追问、质疑或补充。
- `AnswererAgent`：答主侧角色，负责根据答主画像和写作风格生成回应建议，用户可以采用、编辑或忽略。

结合方式：

- 写作页顶部导航里的“直答”可以静态保留，强化知乎原生环境。
- 写作页右侧创作助手内增加一个轻量模块 `直答参照`，只展示写前缺口和参考素材，不触发评论。
- 发布后的回答详情页中，在评论区上方或首条 Agent 评论旁边增加一个轻量入口：`直答追问` 或 `直答补充`。
- 这个入口的作用不是总结全网，而是先由 `DirectAnswerAgent` 判断当前回答的缺口，再由 `AskerAgent` 生成一条真实的评论区追问或补充。
- 视觉上贴近知乎直答：使用知乎蓝/紫蓝小标识、短标题、轻量解释，不做大面积 AI 面板。
- 交互上贴近评论区：点击后直接在评论区出现一条由提问者侧 Agent 发出的追问/补充，而不是打开新的聊天窗口。

推荐命名：

- `直答追问`：强调对当前回答继续追问。
- `直答补充`：强调补足回答没有覆盖的细节。
- `AI 追问`：更直白，但知乎原生感弱一些。

第一版推荐使用 `直答追问`。

它和提问者侧 Agent 的关系：

- `DirectAnswerAgent` 是中立分析者，负责“这个回答还缺什么”。
- `AskerAgent` 是评论区发言者，负责“如果我是提问者/读者，我会怎么追问”。
- `AnswererAgent` 是答主辅助者，负责“答主可以怎么回，才符合自己的风格”。
- `直答追问` 是触发入口和产品包装，不是一个独立评论账号。

示例流程：

```text
答主发布回答
  ↓
进入回答详情页
  ↓
评论区出现“直答追问”入口
  ↓
用户点击
  ↓
DirectAnswerAgent 读取回答、问题、参考资料和评论上下文
  ↓
DirectAnswerAgent 产出一个最值得追问的缺口
  ↓
AskerAgent 把缺口改写成真人评论
  ↓
评论区新增一条真实追问
  ↓
AnswererAgent 在输入框上方给出回应建议
```

这样结合后，比赛表达会更像知乎原生产品能力：不是“我们做了个评论机器人”，而是“知乎直答能力进入回答发布后的评论互动，帮助答主即时看到读者侧的真实追问”。

#### 具体落地形态

写前 `直答参照`：

- 位置：右侧创作助手的 A2A 主卡片下方，和画像、参考素材同层。
- 标题：`直答参照`
- 副标题：`基于知乎相似回答与全网资料，定位这题最容易被追问的点`
- 内容 1：`问题核心`，一句话说明问题真正要解决的矛盾。
- 内容 2：`回答缺口`，列 2-3 个缺口，每个缺口不超过 22 字。
- 内容 3：`参考链接`，展示 2 条知乎相似回答和 1 条全网链接，链接标题使用来源原文标题。
- 操作：`用于写作策略`，点击后把缺口加入写前互搏上下文。

发布后 `直答追问`：

- 位置：回答详情页评论区顶部，位于“写评论”输入框上方或第一条评论前。
- 形态：一条轻量提示条，不做大卡片。
- 文案：`直答发现这个回答还有 1 个读者可能会追问的点`
- 主按钮：`生成追问`
- 次按钮：`查看依据`
- 点击 `查看依据` 展开 3 行依据：回答缺口、关联参考、预计读者意图。
- 点击 `生成追问` 后进入 loading：`直答正在阅读回答...`
- 完成后不展示分析报告，只在评论区追加一条提问者/读者评论。

评论区展示规则：

- 评论账号展示为提问者或读者身份，不展示 `DirectAnswerAgent`。
- 评论内容必须短，像知乎真人评论，避免“根据分析”“综合来看”等报告腔。
- 答主输入框上方出现 `答主 Agent 建议`，默认折叠为一行。
- 用户点击 `采用` 后填入输入框，不自动发送。
- 用户点击 `发送并生成下一轮` 后，再触发 `AskerAgent` 的下一条回复。

一次 `直答追问` 的结构化输出：

```json
{
  "neutral_gap": "没有解释数学基础一般的学生如何判断自己是否适合计算机",
  "evidence": [
    "问题描述中提到'听说学起来很苦，而且要秃头'",
    "相似高赞回答普遍会先拆分数学、编程、信息差三个门槛"
  ],
  "asker_intent": "确认自己是否适合选择这个专业",
  "asker_comment": "我想问下，如果数学基础一般但对计算机挺感兴趣，是不是也能选？还是说这个专业真的很吃天赋？",
  "answerer_suggestion": "先回应他的焦虑，再把'数学基础、持续编码、信息差'分开讲，避免直接劝退。"
}
```

推荐现场演示用例：

```text
题目：对于想选择计算机类专业的新生，有哪些入学前必须了解的「残酷真相」？
答主发布一段回答：计算机并不是只会写代码，真正难的是长期自学和面对变化。
点击：直答追问
评论区新增：那如果我是高三生，数学一般但挺喜欢电脑，这种情况适合选吗？
答主 Agent 建议：不要直接说适合/不适合，先给判断方法。
答主采用并编辑后发送。
提问者 Agent 第二轮：那大学前应该先补数学，还是先学一门编程语言？
```

### 发布后消息通知体系

用户补充的知乎通知截图说明：发布后的 Agent 互动不能只在评论区出现，还要接入知乎式顶部消息提醒。这样演示时更像真实知乎产品，而不是一个孤立的 Demo 页面。

设计原则：

- 通知是“入口”和“提醒”，不是 Agent 工作台。
- 真正的互动内容仍然发生在回答详情页/文章页评论区。
- 通知里的 `Agent 对话` tag 表示这条通知来自提问者侧 Agent 的 A2A 互动，不表示 DirectAnswerAgent 是一个发言账号。
- 通知里不展示“DirectAnswerAgent 分析报告”，只展示知乎式互动文案。
- 通知红点和数字只用于 Demo 演示，可以静态或由本地状态驱动。
- 通知下拉层视觉参考截图：右上角白色浮层、顶部三段图标 tab、列表分割线、底部“设置/查看全部通知”。

通知入口位置：

- 顶部导航右侧保留 `消息`、`私信`、`创作中心`、头像。
- `消息` 图标可以显示红色未读角标，例如 `1` 或 `6`。
- 用户点击 `消息` 后，右侧弹出通知下拉层。
- 下拉层宽度约 360-420px，右对齐顶部导航，不遮挡主要回答内容太多。

通知下拉 tab：

截图中红框区域可简化为三个 tab：

- `互动`：评论、赞同、收藏、感谢、直答追问等内容互动。
- `关注`：关注者、被关注、邀请回答等社交关系。
- `赞同`：赞同/喜欢类提醒。

第一版 Demo 只需要让 `互动` tab 真实更新，另外两个 tab 可以静态展示或置灰。

通知事件类型：

1. 发布后提问者首评

```text
提问者 小青 评论了你的回答「对于想选择计算机类专业的新生，有哪些入学前必须了解的残酷真相？」
“那如果我数学一般，但是对计算机很感兴趣，还适合选吗？”
刚刚
```

2. Agent 直答追问

```text
Agent 对话 · 直答追问
提问者小青 追问了你的回答「对于想选择计算机类专业的新生，有哪些入学前必须了解的残酷真相？」
“大学前应该先补数学，还是先学一门编程语言？”
刚刚
```

注意：这里的 `Agent 对话` 是通知 tag，用来告诉答主“这是提问者 Agent 发起的 A2A 互动”。`直答追问` 是触发能力名。点击后不要在通知浮层里继续对话，而是跳转到 1:1 复刻的知乎回答/文章详情页，在详情页评论区继续互动。评论区里仍只出现提问者/读者身份，不出现 DirectAnswerAgent。

3. 答主回复后的提问者再互动

```text
小青 回复了你在评论区的解释
“这样说我清楚多了，不过如果只剩两个月准备，优先做哪件事？”
刚刚
```

4. 普通知乎氛围通知

用于让页面更真实，可以静态混入 2-3 条：

```text
知乎用户 54321、小青 赞同了你的文章《娱乐至死》有感，我们正在被时代驯化吗
炤眼 赞同了你的想法 #知乎黑客松 2026 #知乎观察
zhuox2 赞同了你的回答 大一到底要怎样做才能兼顾学习与社交？
```

通知和评论区联动：

- 当 `publish_feedback_ready` 到达时，评论区追加提问者首评，同时消息角标 +1，通知下拉新增一条“评论了你的回答”。
- 当 `zhida_followup_ready` 到达时，评论区追加提问者 Agent 的追问，同时消息角标 +1，通知下拉新增一条带 `Agent 对话` tag 的“直答追问”通知。
- 当 `comment_turn_ready` 到达时，评论区追加下一轮提问者回复，同时消息角标 +1。
- 用户点击通知项后，如果仍在编辑页或通知浮层所在页面，则先跳转到发布后的知乎回答/文章详情页，再滚动到对应评论，并短暂高亮该评论。
- 用户打开通知下拉后，可以将未读角标清零，但评论区内容不变。

通知数据结构建议：

```json

  "type": "comment",

  "actor_name": "小青",

  "created_at_label": "刚刚",

```

```json
{

  "type": "zhida_followup",
```

演示路径：

```text
答主发布回答
  ↓
评论区出现提问者首评
  ↓
顶部“消息”出现红点 1
  ↓
点击消息，出现知乎式通知下拉
  ↓
点击带“Agent 对话”tag 的直答追问通知
  ↓
跳转到 1:1 复刻的知乎回答/文章详情页
  ↓
页面滚动到提问者 Agent 的追问评论
  ↓
答主在文章详情页评论区继续回复
```

第一版实现范围：

- 必须复刻顶部消息入口和下拉层视觉。
- 必须支持发布后新增通知。
- 必须支持 `Agent 对话` tag。
- 必须支持点击 Agent 通知跳转到发布后的回答/文章详情页并定位评论。
- `关注`、`赞同` tab 可以静态。
- 不需要接入真实知乎通知接口。

## Agent 编排方案

### 编排目标

Agent 编排要同时满足三个目标：

- 真实：答主画像和素材来源尽量来自知乎开放接口，LLM 负责分析和归纳，不负责凭空造数据。
- 快：写前建议尽量在 30 秒内出现，不用多轮串行 LLM 模拟“聊天”。
- 有用：输出必须服务创作者决策，包括写不写、怎么写、引用什么、避开什么、发布后可能被怎样追问。

### 总体 DAG

编排采用“并行采集 + 并行画像 + 单次合并互搏 + 发布后页面轻量互动”的 DAG。

```text
创建 Demo session
        |
        v
并行数据采集
  ├─ 答主关注流 / 关注列表 / 粉丝列表
  ├─ 知乎相似问题 / 相似回答 / 高赞回答
  ├─ 全网搜索结果或静态参考素材
  └─ 提问者模拟模板
        |
        v
并行 Agent 分析
  ├─ AnswererProfileAgent
  ├─ AskerProfileAgent
  └─ ResearchAgent
        |
        v
DebateAndStrategyAgent
  ├─ 合并两轮互搏
  └─ 生成写作策略
        |
        v
前端展示写前建议
        |
        v
答主编辑 / 模拟发布
        |
        v
切换到回答详情/结果页
        |
        v
PostAnswerInteractionAgent
  ├─ 生成提问者首条评论
  └─ 评论区短轮次互动
```

### 阶段 0：Session 初始化

触发：

- 用户打开 Demo 页面，或点击创作助手中的 `A2A 创作协同`。

后端动作：

- 创建 session。
- 记录当前问题 fixture。
- 记录提问者 persona 模板。
- 初始化 answer draft 为空。
- 初始化 Agent 状态。

前端展示：

- 创作助手入口卡片显示 `开始分析`。
- 进入 A2A 子面板后，步骤列显示全部待处理。

### 阶段 1：并行数据采集

并行任务：

- `AnswererContextFetcher`
  - 调用 `GET /openapi/feed/following`。
  - 调用 `GET /openapi/user/following`。
  - 调用 `GET /openapi/user/followers`。
  - 输出原始关注流、关注对象、粉丝概览。
- `ZhihuReferenceFetcher`
  - 基于当前问题标题调用知乎搜索或比赛提供的相似内容接口。
  - 拉取若干相似问题、相似回答、高赞回答摘要。
  - 输出带链接的参考候选。
- `WebReferenceFetcher`
  - 如果有真实全网搜索能力，拉取 3-5 条相似内容。
  - 如果没有，使用静态兜底素材，并标记来源类型。
- `AskerPersonaLoader`
  - 加载提问者侧 Demo fixture，例如焦虑求助型。

输出：

- `answerer_raw_context`
- `zhihu_reference_candidates`
- `web_reference_candidates`
- `asker_fixture_context`

前端展示：

- 步骤列显示 `正在读取你的关注流`。
- 如果知乎接口成功，显示“已读取关注流/关注列表/粉丝列表”。
- 如果知乎接口失败，显示降级提示，但继续后续流程。

### 阶段 2：并行 Agent 分析

三个 Agent 并行跑，减少等待。

#### AnswererProfileAgent

输入：

- `answerer_raw_context`
- 当前问题标题和描述
- 可选草稿

LLM 输出结构：

- `interest_domains`：兴趣领域，必须来自关注流或关注列表证据。
- `audience_expectation`：粉丝/受众可能期待。
- `writing_advantages`：当前问题上答主可用优势。
- `style_constraints`：适合的表达方式。
- `fabrication_boundaries`：不能假装拥有的经历。
- `evidence`：每条结论对应的接口数据证据摘要。

设计要求：

- 画像要回答“这对我写这题有什么用”，而不是做人格测试。
- 每条结论尽量短，前端只展示 2-3 条最有用的。

#### AskerProfileAgent / 提问者侧 Agent 初始化

输入：

- `asker_fixture_context`
- 当前问题标题和描述

LLM 输出结构：

- `persona_type`：焦虑求助型/理性质疑型/经验补充型。
- `core_need`：真正想解决的问题。
- `rejection_points`：讨厌的回答方式。
- `trust_triggers`：什么内容会让 TA 感到可信。
- `likely_first_comment`：发布后可能评论的方向，不直接作为最终评论使用。

设计要求：

- 提问者侧数据可以是 Demo fixture，但 Agent 互动必须是真实运行，不要写成“静态分析报告”。
- 提问者侧 Agent 应围绕问题和 persona 一致，不要变成万能用户。
- UI 中 persona 标签使用知乎蓝灰体系，不用黄色。

#### ResearchAgent

输入：

- `zhihu_reference_candidates`
- `web_reference_candidates`
- 已有回答摘要

LLM 输出结构：

- `reference_links`：带链接参考素材。
- `high_vote_patterns`：高赞回答常用思路。
- `overused_angles`：已经说烂的角度。
- `fresh_angles`：可切入的新角度。
- `risk_points`：争议和误导风险。

设计要求：

- `reference_links` 必须尽量保留真实 URL。
- 如果某条素材来自静态兜底，UI 要显示“演示素材”而不是伪装真实搜索结果。

### 阶段 3：合并互搏与策略生成

为了压缩等待时间，不让三个 Agent 真的串行聊天 6 次。

使用 `DebateAndStrategyAgent` 一次 LLM 调用完成：

- 根据三个 Agent 的结构化结果组织两轮互搏。
- 生成可展示的互搏日志。
- 生成最终写作策略。

输入：

- `answerer_profile`
- `asker_profile`
- `research_brief`
- 当前问题

输出结构：

- `round_1_positions`
  - 答主画像视角：我适合怎么答。
  - 提问者画像视角：TA 需要什么。
  - 全网研究视角：外部语境是什么。
- `round_2_challenges`
  - 每个视角挑战另一个视角的盲点。
- `strategy`
  - `should_answer`：建议回答/谨慎回答/不建议回答。
  - `primary_angle`：主切入角。
  - `outline`：回答结构，不生成完整正文。
  - `avoid`：避坑点。
  - `reference_links`：建议参考的链接。
  - `expected_followups`：发布后可能被追问的问题。

前端展示：

- 默认只展示 `strategy` 的最有用三项。
- `round_1_positions` 和 `round_2_challenges` 折叠在“查看互搏过程”里。
- `reference_links` 展示为可点击列表，标题 + 来源 + 可借鉴点。

### 阶段 4：写作中辅助

答主编辑草稿时，不重新跑完整互搏。

可选轻量能力：

- 字数和结构检测本地完成。
- 如果用户点击“检查当前草稿”，只调用一个短 LLM，对草稿和策略做对照。

输出：

- 是否偏离策略。
- 是否踩到提问者禁忌。
- 哪一段需要补证据。

### 阶段 5：发布后进入回答详情页

触发：

- 用户点击底部 `发布回答` 或 Demo 的 `模拟发布`。
- 用户点击顶部消息下拉中的 `Agent 对话 · 直答追问` 通知。

前端动作：

- 退出回答编辑器。
- 进入单独的发布结果/回答详情页，视觉按知乎文章/回答详情页 1:1 复刻。
- 展示已发布回答正文。
- 展示评论区容器。
- 在评论区展示“提问者 Agent 正在阅读回答...”的短状态。
- 如果入口来自通知，进入详情页后定位到对应评论锚点。

后端动作：

- 使用 `PostAnswerInteractionAgent` 一次短 LLM 调用，生成提问者第一条评论。

输入：

- 答主草稿。
- `asker_profile`
- `strategy.expected_followups`
- 已有评论上下文。

输出结构：

- `asker_comment`
  - 语气像知乎评论区真人。
  - 可以认可、追问、反驳或补充。
- `comment_intent`
  - 认可/追问/质疑/补充。

前端展示：

- 提问者评论出现在发布后页面的评论区。
- 不在创作助手中展示发布反馈卡片。
- 不默认展示答主回应建议。

### 阶段 6：评论区多轮互动

每一轮只调用 `PostAnswerInteractionAgent`，不回到完整画像和互搏流程。

输入：

- 最近评论线程。
- 答主最新回复。
- 提问者画像。
- 写前策略。

输出：

- 提问者下一条回复。

限制：

- 默认最多 2-3 轮。
- 每次输出短，避免现场等待。
- 如果 LLM 超时，保留已有评论并允许重试。

直答追问触发：

- 如果用户点击发布后页面的 `直答追问`，等同于触发一轮评论区互动。
- 与普通答主手动回复不同，`直答追问` 不要求用户先输入回复。
- Agent 会基于当前回答中最容易被追问的一点，主动生成一条提问者侧评论。
- 这条评论仍进入同一评论流，后续答主可以继续手动回复。

### 状态机

```text
idle
  -> collecting_context
  -> profiling
  -> researching
  -> debating
  -> strategy_ready
  -> drafting
  -> published
  -> comment_turn_running
  -> comment_turn_ready
```

错误状态：

- `context_degraded`：知乎接口失败，使用最小画像继续。
- `llm_retryable_error`：当前 LLM 阶段失败，可重试。
- `interaction_timeout`：评论互动超时，可重试本轮。

### SSE 事件与前端映射

- `profile_fetching`：步骤列点亮“答主画像”。
- `answerer_profile_ready`：展示真实接口画像摘要和证据。
- `asker_profile_ready`：展示蓝灰色提问者 persona 标签。
- `research_ready`：展示参考素材和高赞思路链接。
- `debate_round`：追加互搏日志。
- `strategy_ready`：展示写作策略，入口卡片显示 `策略已就绪`。
- `publish_feedback_ready`：发布后页面评论区追加提问者侧 Agent 第一条评论。
- `comment_turn_ready`：评论区追加下一轮提问者回复。

## 完整交互流程

1. 用户打开 localhost Demo 页面。
2. 页面展示模拟知乎回答编辑页。
3. 后端创建 Demo session。
4. 后端真实调用知乎开放接口，获取当前登录答主的关注流、关注列表、粉丝列表。
5. 后端基于提问者侧 Demo fixture 创建可互动画像。
6. 后端获取知乎搜索或静态研究语境。
7. 答主画像 Agent、提问者画像 Agent、全网研究 Agent 并行分析。
8. 互搏编排 Agent 将三方观点整合成两轮互搏日志和写作策略。
9. 前端右侧面板展示写前建议。
10. 答主在编辑器里写回答，或插入 Demo 草稿。
11. 答主点击“发布回答/模拟发布”。
12. 前端切换到发布后的回答详情/结果页。
13. 提问者侧 Agent 真实调用大模型，在评论区生成第一条评论。
14. 答主可以在评论区手动回复。
15. 提问者侧 Agent 继续生成下一轮评论区回复。
16. Demo 展示“写前判断 + 发布后评论区互动”的完整闭环。

## 低等待时间要求

用户明确要求：大模型调用必须真实，但交互时间要压缩，不能等很久。

目标：

- 首次有用写作建议尽量控制在 30 秒以内。
- 页面任何阶段不能像卡死，必须展示明确进度。
- 评论区每轮回复要短，避免现场演示等待过长。

策略：

- 知乎接口并行请求，不串行等。
- 答主画像 Agent、提问者画像 Agent、全网研究 Agent 并行跑。
- 两轮互搏尽量合并成一个结构化 LLM 调用，而不是 6 次来回对话。
- 输出严格限长，避免长篇生成。
- 写前阶段缓存答主画像，本 session 内复用。
- 发布后评论区互动复用写前结论，不重新跑完整互搏。
- 评论区多轮只调用发布后页面互动 Agent。
- 前端用进度状态、阶段日志、骨架屏展示进展。
- 如果非关键接口失败，降级展示，不阻塞整条链路。

建议第一版 LLM 调用预算：

- 1 次答主画像分析。
- 1 次提问者画像扩写。
- 1 次全网研究总结。
- 1 次合并互搏 + 写作策略。
- 1 次发布后提问者评论。
- 每轮评论互动 1 次短调用。

## 后端接口设计

写前分析进度使用 SSE，不再使用 WebSocket。

原因：

- 当前链路主要是服务端向浏览器推送阶段进度和 LLM 结果，双向长连接不是刚需。
- 用户动作可以用普通 REST 表达，例如发布、评论回复、直答追问。
- SSE 基于 HTTP，前端 `EventSource` 原生支持，实现和调试成本更低。
- 断线自动重连和代理兼容性更适合比赛 Demo。
- 如果未来需要真正的多人实时协作或高频双向通信，再考虑 WebSocket 或 WebTransport。

建议 REST：

- `POST /api/demo/session/start`：创建 Demo session，启动写前分析。
- `GET /api/demo/session/{session_id}`：获取当前 session 状态。
- `GET /api/demo/session/{session_id}/events`：SSE 事件流，推送写前分析进度和结果。
- `POST /api/demo/session/{session_id}/publish`：提交答主草稿，进入发布后页面并生成提问者首条评论。
- `POST /api/demo/session/{session_id}/comment`：提交答主评论回复，生成下一轮互动。
- `POST /api/demo/session/{session_id}/zhida-followup`：发布后页面点击“直答追问”，无需答主输入，生成一条提问者侧追问。

建议 SSE 消息类型：

- `connected`：事件流已连接，写前分析开始。
- `profile_fetching`：正在拉取答主关注流和关注关系。
- `profiling`：正在进行画像和语境分析。
- `answerer_profile_ready`：答主画像完成。
- `asker_profile_ready`：提问者画像完成。
- `research_ready`：问题语境研究完成。
- `debate_round`：互搏日志更新。
- `strategy_ready`：写作策略可用。
- `publish_feedback_ready`：发布后页面首条提问者评论完成。
- `comment_turn_ready`：评论区下一轮反馈完成。
- `zhida_followup_ready`：直答追问生成完成，评论区追加一条提问者侧追问。
- `error`：可恢复或不可恢复错误。

## 前端设计

### 主编辑区

主编辑区要 1:1 复刻截图中的知乎回答编辑页，而不是普通表单。

包含：

- 顶部知乎导航。
- 顶部消息入口、私信入口、创作中心入口、头像。
- 问题标题。
- 问题描述。
- 关注/邀请回答等静态动作。
- 已有回答摘要。
- 图文回答编辑器。
- 全屏编辑入口和全屏编辑状态。
- 知乎式编辑器工具栏。
- 插入 Demo 草稿按钮。
- 模拟发布按钮。
- 发布后跳转/切换到回答详情页。

### 顶部消息通知下拉

消息通知是发布后互动承接的一部分，视觉参考用户提供的通知截图。

包含：

- 顶部导航里的 `消息` 图标和未读红点。
- 点击后出现右侧浮层。
- 浮层顶部是三枚图标 tab：互动、关注、赞同。
- 默认打开 `互动` tab。
- 列表展示知乎式通知文本，不展示 Agent 内部字段。
- 底部保留 `设置` 和 `查看全部通知`。

通知内容来源：

- Demo 初始静态通知 3-4 条，用来复刻真实知乎氛围。
- 发布后提问者首评新增一条通知。
- 直答追问新增一条通知。
- 评论区下一轮提问者回复新增一条通知。

点击行为：

- 点击普通静态通知不必跳转，可以保持下拉层。
- 点击带 `Agent 对话` tag 的通知，跳转到 1:1 复刻的知乎回答/文章详情页，并滚动到对应评论。
- 打开通知下拉后可清除红点，但保留列表中的通知。

### 右侧创作助手面板

右侧面板必须模拟知乎编辑器里的“创作助手”，展示 Agent 工作，不要只是 loading。

包含：

- 创作助手标题栏：标题、收起或关闭按钮、状态提示。
- 静态功能入口：内容检测、智能排版、提取导语、AI 配图、更多功能。
- A2A 核心入口：`A2A 创作协同`，作为新增功能卡片出现在创作助手首页。
- 答主画像卡片。
- 提问者画像卡片。
- 全网研究卡片。
- A2A 两轮互搏日志。
- 写作策略卡片。
- 发布后页面入口状态。

创作助手首页状态：

- 第一屏保留原生功能氛围，不直接铺满复杂分析结果。
- `A2A 创作协同` 是唯一明显的新增能力入口。
- 原生功能块可以静态不可用，但 hover 或点击应表现为普通功能入口，不抢主线。

A2A 子面板状态：

- 顶部显示返回入口和当前阶段。
- 写前阶段优先展示“最有用结论”，画像和互搏日志可以折叠。
- 不包含发布反馈 tab。
- 不包含评论互动输入框。
- 所有状态都要短文案、低密度，不做长报告页。

### 演示页状态

至少覆盖这些状态：

- 初始化。
- 创作助手首页。
- A2A 创作协同入口未开始。
- 拉取答主数据中。
- 画像分析中。
- 互搏中。
- 写作建议已生成。
- 草稿编辑中。
- 已模拟发布。
- 发布后回答详情页。
- 直答追问生成中。
- 发布后评论区多轮互动中。
- 失败可重试。

## 错误与降级

知乎接口失败：

- UI 明确提示“真实知乎接口调用失败，已使用最小模拟画像继续演示”。
- 不阻塞 LLM 和页面演示。

LLM 调用失败：

- 显示失败阶段。
- 提供“重试本阶段”。
- 已完成的画像和研究结果保留。

LLM 调用过慢：

- UI 显示当前阶段和耗时。
- 允许用户继续在编辑器中写草稿。
- 如果策略还没返回，右侧展示“可先写草稿，策略生成后会补充”。

发布后页面互动失败：

- 已发布回答保留。
- 评论区显示重试按钮。
- 不回滚前面的写作策略。

## 与旧实现的差异

旧实现：

- 输入知乎问题链接。
- 采集/过滤数据。
- 用户 Agent 和答主 Agent 对话 3 轮。
- 生成需求对齐看板。
- 生成写作框架。

新实现：

- 不再以“输入问题链接”为主入口。
- 入口是模拟知乎回答编辑页。
- A2A 能力嵌入编辑器右侧“创作助手”区域，而不是独立外置面板。
- 答主是真实登录人画像。
- 提问者是模拟画像。
- A2A 主体变成答主画像 Agent、提问者画像 Agent、全网研究 Agent。
- 互搏目标不是“达成需求共识”，而是形成可供答主参考的写作策略。
- 写作策略之后，编辑页阶段结束；发布后评论区互动发生在单独的回答详情/结果页。

## 非目标

第一版不做：

- 真实发布到知乎。
- 完整浏览器插件。
- 完整复刻知乎所有页面。只 1:1 复刻比赛 Demo 所需的回答编辑页。
- 自动生成并发布完整回答正文。
- 真实提问者 user_id 画像。
- 无限轮评论区对话。

## 验收标准

比赛 Demo 应该能展示：

- 打开页面后像是在知乎写回答，且关键布局 1:1 对齐参考截图。
- 编辑器包含普通模式和全屏编辑模式。
- A2A 功能位于编辑器右侧“创作助手”内，不是独立外置面板。
- 答主画像来自真实知乎开放接口。
- 大模型真实生成画像、互搏、策略和发布后评论。
- 提问者侧数据即使使用 Demo fixture，评论区互动也必须是真实 Agent 运行结果，不是静态脚本或预测报告。
- 写作建议不是泛泛模板，而能体现答主视角、提问者视角和全网语境。
- 点击模拟发布后，页面切换到发布后的回答详情/结果页。
- 提问者侧 Agent 在发布后页面评论区自动给出第一条评论。
- 发布后页面提供 `直答追问` 入口，点击后能生成一条提问者侧追问并进入评论流。
- 答主能在发布后页面继续评论区互动，至少跑 1-2 轮。
- 等待时间可控，页面不会长时间无反馈。
- 明确传达“AI 辅助，人为主导，不代写完整正文”。
